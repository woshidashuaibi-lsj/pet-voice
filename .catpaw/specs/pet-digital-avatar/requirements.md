# Live2D 宠物分身 - 需求文档

## 简介

为毛孩子翻译官添加宠物数字分身页面，通过 Live2D 动画技术展示宠物的实时状态，支持点击互动、情绪响应、数据进化等功能。用户留存对话和打卡数据越多，宠物分身越立体生动。

## 术语表

| 术语 | 定义 |
|------|------|
| **avatar_state** | 宠物分身状态快照，包含四维数据、进化等级、当前心情等 |
| **四维数据** | contentment(满足度) / energy(能量) / affection(亲密度) / health(健康度) |
| **进化等级** | 1-5 级，基于总对话轮数累积 |
| **交互热区** | Live2D 模型上可点击的区域（头、背、身体、尾巴） |
| **moodScore** | 综合情绪评分 (0-100)，由四维数据加权计算 |
| **personality** | 宠物性格特征（来自 pets.memory 数据） |
| **recordInteraction** | 记录用户与宠物的每次点击交互 |

## 需求规范

### Requirement 1：数据模型设计与存储

**用户故事**：作为系统设计者，我需要定义宠物分身的数据结构，以便存储和管理宠物的进化状态。

#### 接受准则

1. **WHEN** 用户首次打开宠物分身页面，**THEN** 系统 **SHALL** 自动为该宠物创建一个 `pet_avatar_states` 文档，包含四维数据、进化等级、性格标签、已解锁动作等信息

2. **WHILE** 宠物分身页面打开时，**THEN** 系统 **SHALL** 每天自动（或用户打开时）调用 `updateAvatarState` 云函数重新计算四维数据

3. **WHEN** 新对话完成且信息被提取时，**THEN** 系统 **SHALL** 自动更新宠物的性格标签，将结果同步到 `pet_avatar_states.personality` 中

4. **WHEN** 用户的对话轮数累积达到阈值（10/50/100/200），**THEN** 系统 **SHALL** 自动升级宠物进化等级并解锁新的动作库

5. **WHERE** 四维数据计算时，**THEN** 系统 **SHALL** 基于以下公式独立计算，并确保最终分数在 0-100 范围内：
   - contentment = (打卡频率 * 0.4 + 对话热度 * 0.3 + 游戏参与 * 0.3)
   - energy = 活跃特征基础分 +/-衰减因子
   - affection = (互动频率 * 0.3 + 对话深度 * 0.4 + 一致性 * 0.3)
   - health = (打卡健康评分 * 0.6 + 无不适占比 * 0.4)

---

### Requirement 2：Live2D 模型集成与动画系统

**用户故事**：作为用户，我想看到一个活生生的 2D 宠物动画，随着互动变得更生动活泼。

#### 接受准则

1. **THE** Live2D 渲染层 **SHALL** 支持至少 15 个不同的动作（待机、睡眠、开心、难受、好奇、调皮、被摸、惊吓、兴奋、疲倦、困惑、进食、整理、亲昵、胜利）

2. **WHEN** 用户打开宠物分身页面时，**THEN** 系统 **SHALL** 自动播放欢迎动作（从 happy / curious / playful 中随机选择），并在 2 秒内完成

3. **WHEN** 页面无用户交互超过 5 秒，**THEN** 系统 **SHALL** 自动播放待机或睡眠相关的循环动作

4. **WHILE** Live2D 动画播放中，**THEN** 系统 **SHALL** 保证动画帧率 ≥ 30 FPS，不卡顿

5. **WHERE** 宠物性格标签包含"活泼"或"好动"时，**THEN** 系统 **SHALL** 提高 playful / excited 等积极动作的播放权重

6. **WHEN** 时间是晚上 22:00 - 早上 08:00，**THEN** 系统 **SHALL** 优先播放睡眠相关动作，即使用户进行交互

---

### Requirement 3：点击互动系统与响应逻辑

**用户故事**：作为用户，我想能够通过点击宠物的不同部位来和它互动，看到它做出不同的反应。

#### 接受准则

1. **WHEN** 用户在 Live2D 模型上点击，**THEN** 系统 **SHALL** 正确识别点击区域（head / back / body / tail），并在 300ms 内触发对应的动作

2. **WHEN** 用户点击头部时，**THEN** 系统 **SHALL** 播放"好奇"或"被摸"动作，并有 70% 概率成功（可失败）

3. **WHEN** 用户点击后背时，**THEN** 系统 **SHALL** 播放"被摸"或"开心"动作，成功概率 ≥ 85%，并增加亲密度 +5 / 满足度 +3

4. **WHEN** 用户双击宠物时，**THEN** 系统 **SHALL** 播放特殊动作"兴奋"，增加亲密度 +10 / 满足度 +8，并显示心形粒子特效

5. **WHEN** 用户长按宠物 ≥ 3 秒时，**THEN** 系统 **SHALL** 播放"亲昵"动作，增加亲密度 +15 / 满足度 +10 / 健康度 +5，并显示光晕特效

6. **WHERE** 宠物当前能量值 < 30 或健康度 < 40 时，**THEN** 系统 **SHALL** 降低点击响应的积极程度，改为"疲倦"或"难受"动作

7. **WHEN** 用户进行点击交互，**THEN** 系统 **SHALL** 在宠物所在位置显示随机的对话气泡，内容包括"你摸我干嘛呢？"、"舒服~"等，并在 2 秒后消失

8. **WHILE** 点击交互频繁时，**THEN** 系统 **SHALL** 实施冷却机制，不同区域的冷却时间为 300-600ms，防止用户无限频繁点击

---

### Requirement 4：情绪到表现的映射系统

**用户故事**：作为设计者，我需要建立四维数据到视觉表现的映射，使宠物的外观能够反映其当前状态。

#### 接受准则

1. **WHEN** moodScore（综合情绪评分）计算完成，**THEN** 系统 **SHALL** 将分数映射到 5 个情绪段级（very_sad / sad / neutral / happy / very_happy）

2. **WHEN** moodScore 处于 81-100 区间，**THEN** 宠物 **SHALL** 展现"very_happy"状态，包括：明亮的皮肤色调、闪闪发光的眼神、快速摇晃的尾巴、挺胸跳跃的姿态

3. **WHEN** moodScore 处于 0-20 区间，**THEN** 宠物 **SHALL** 展现"very_sad"状态，包括：灰暗的皮肤色调、无神的眼睛、下垂的尾巴、蜷缩的姿态

4. **WHEN** 宠物进化等级升级时，**THEN** 系统 **SHALL** 立即播放"胜利"动作，并显示升级动画和提示文案

5. **WHERE** 健康度 < 50 时，**THEN** 系统 **SHALL** 自动触发"难受"相关的动作库权重提升

---

### Requirement 5：数据持久化与交互记录

**用户故事**：作为系统，我需要记录每次用户与宠物的交互，用于后续的数据分析和奖励计算。

#### 接受准则

1. **WHEN** 用户完成一次点击交互，**THEN** 系统 **SHALL** 异步调用 `recordInteraction` 云函数记录此次交互，包括交互类型、时间戳、维度变化等

2. **WHEN** 云函数记录交互时，**THEN** 系统 **SHALL** 同时更新 `pet_avatar_states.stats.totalInteractions` 和相关维度值

3. **WHILE** 交互记录进行中，**THEN** 前端 **SHALL** 不阻塞 UI 交互，用户可以继续点击或导航

4. **WHEN** 单日点击交互超过 100 次，**THEN** 系统 **SHALL** 对维度增长实施上限，避免数据不合理增长

5. **WHEN** 连续 7 天每天都有交互时，**THEN** 系统 **SHALL** 记录"连续活跃天数"并可能触发成就或奖励

---

### Requirement 6：UI/UX 与视觉反馈

**用户故事**：作为用户，我希望看到清晰、直观的界面和即时的反馈效果。

#### 接受准则

1. **THE** 宠物分身页面 **SHALL** 由以下主要组成部分构成：
   - 顶部导航栏（返回、标题、分享按钮）
   - 中央 Live2D 模型渲染区域（占屏幕高度 50-60%）
   - 四维数据条显示（contentment / energy / affection / health）
   - 进化等级和升级进度条
   - 底部功能按钮（小游戏、礼物、回顾等）

2. **WHEN** 用户点击宠物并产生维度变化时，**THEN** 系统 **SHALL** 显示浮起的数字（+5 affection）并伴随变色动画

3. **WHEN** 四维数据条更新时，**THEN** 系统 **SHALL** 使用平滑的过渡动画（300-500ms），而非立即跳转

4. **WHEN** 页面加载完成，**THEN** 系统 **SHALL** 在 1 秒内展示宠物分身（包括 Live2D 加载）

5. **WHILE** Live2D 模型加载中，**THEN** 系统 **SHALL** 显示加载动画或占位符，并提示用户"宠物正在赶来..."

---

### Requirement 7：与现有功能的联动

**用户故事**：作为产品，我需要确保宠物分身功能与现有的对话、打卡系统无缝协同。

#### 接受准则

1. **WHEN** 用户在健康咨询/训练页面完成一次对话，**THEN** `avatar_state` 中的对话计数应在 24 小时内更新

2. **WHEN** 用户完成今日打卡，**THEN** 系统 **SHALL** 立即在宠物分身页面可见反馈（如健康度 +5）

3. **WHEN** 系统从 `pets.memory` 中提取到新的性格标签，**THEN** 这些标签应在下次打开宠物分身页面时生效，影响互动响应

4. **WHEN** 用户解锁新的装饰品或动作时，**THEN** 宠物分身页面应实时显示已解锁内容

---

### Requirement 8：性能与兼容性

**用户故事**：作为开发者，我需要确保功能在各类手机上都能流畅运行。

#### 接受准则

1. **THE** Live2D 渲染 **SHALL** 在低端手机（骁龙 630 及以上）上保持 ≥ 30 FPS

2. **THE** 宠物分身页面首次加载时间 **SHALL** ≤ 2 秒（包括 Live2D 模型加载）

3. **THE** Live2D 模型文件大小 **SHALL** ≤ 2MB（单个模型）

4. **WHILE** 点击交互频繁时，**THEN** 内存占用 **SHALL** ≤ 100MB

5. **THE** 功能 **SHALL** 在 iOS 和 Android 微信小程序上都正常运行

---

## 核心价值

这个功能的核心是**情感连接的可视化**：
- 用户看到对话越多，宠物分身越懂它
- 打卡越勤快，宠物越活力满满
- 互动越频繁，亲密度越高
- 这形成了一个"养成游戏"般的心理激励

最终让用户感受到：**这不只是工具，而是我宠物的数字孪生体。**
