### 核心设计原则

| 原则 | 说明 |
|------|------|
| 前后端分离 | 小程序端只负责 UI 渲染与交互，业务逻辑全部在服务端 |
| 微服务拆分 | 用户、宠物、AI 分析三个独立服务，可独立扩缩容 |
| 异步处理 | 音频分析任务进消息队列，避免接口超时 |
| 数据安全 | 音频文件加密存储，传输全程 HTTPS，用户数据脱敏 |

#### 3.2.2 录音按钮组件关键代码

```javascript
// components/record-button/index.js
const recorderManager = wx.getRecorderManager()

Component({
  data: {
    isRecording: false,
    duration: 0,
    minDuration: 3,   // 最短录音秒数
    maxDuration: 15,  // 最长录音秒数
  },
  methods: {
    onTouchStart() {
      // 请求麦克风权限
      wx.authorize({ scope: 'scope.record' }).then(() => {
        this.startRecord()
      }).catch(() => {
        wx.showToast({ title: '请授权麦克风权限', icon: 'none' })
      })
    },
    startRecord() {
      this.setData({ isRecording: true, duration: 0 })
      this._timer = setInterval(() => {
        this.setData({ duration: this.data.duration + 1 })
        if (this.data.duration >= this.data.maxDuration) {
          this.stopRecord()
        }
      }, 1000)
      recorderManager.start({
        duration: this.data.maxDuration * 1000,
        format: 'mp3',
        sampleRate: 44100,
        numberOfChannels: 1,
        encodeBitRate: 96000,
      })
    },
    stopRecord() {
      clearInterval(this._timer)
      this.setData({ isRecording: false })
      if (this.data.duration < this.data.minDuration) {
        wx.showToast({ title: `请至少录音 ${this.data.minDuration} 秒`, icon: 'none' })
        recorderManager.stop()
        return
      }
      recorderManager.stop()
    },
    onTouchEnd() {
      if (this.data.isRecording) this.stopRecord()
    },
  },
  lifetimes: {
    attached() {
      recorderManager.onStop((res) => {
        this.triggerEvent('recorded', { tempFilePath: res.tempFilePath })
      })
    }
  }
})
```

## 四、后端服务设计

### 4.1 用户服务（Node.js + Koa2）

**职责**：微信登录鉴权、用户信息管理、会员状态管理、Token 发放

#### 登录流程

```
小程序调用 wx.login() → 获取临时 code
    │
    ▼
POST /api/user/login { code }
    │
    ▼
服务端调用微信 jscode2session 接口
获取 openid + session_key
    │
    ▼
查询/创建用户记录（users 表）
生成 JWT Token（有效期 7 天）
    │
    ▼
返回 { token, userInfo, petList }
```

#### Token 刷新策略

- Access Token 有效期：**7天**
- 每次请求在响应头带回剩余有效时间
- 有效期不足 1 天时，自动在响应头下发新 Token
- 小程序端 `request.js` 拦截器自动更新本地存储

### 4.2 宠物服务（Node.js + Koa2）

### 4.3 AI 分析服务（Python + FastAPI）

**职责**：接收音频文件、音频预处理、模型推理、结果结构化、写入结果存储

#### 处理流程

```
接收音频上传请求（或从 OSS 下载）
    │
    ▼
音频预处理
├── 格式转换（统一为 WAV 16kHz mono）
├── 降噪（noisereduce）
├── 有效音段检测（VAD）
└── 特征提取（MFCC + 梅尔频谱图）
    │
    ▼
模型推理
├── 宠物种类识别（猫/狗分类器）
├── 情绪分类（8类情绪标签 + 置信度）
└── 异常检测（是否含健康预警特征）
    │
    ▼
LLM 文案生成
├── 构建 Prompt（含宠物档案信息 + 情绪标签 + 置信度）
└── 调用 GPT-4o / 通义千问生成自然语言解读
    │
    ▼
结构化结果写入 MySQL + Redis（缓存 1 小时）
更新任务状态为 completed
```

### 4.4 异步任务设计

音频分析耗时约 **1-5秒**，使用消息队列异步处理，避免接口超时：

```
客户端 POST /api/analysis/submit
    │
    ▼
后端验证 → 上传音频到 OSS → 创建 analysis_tasks 记录（status: pending）
    │ 返回 taskId（立即响应）
    ▼
投递消息到 RocketMQ 队列
    │
    ▼
AI 服务消费队列 → 执行分析 → 更新 analysis_tasks（status: completed）
    │
    ▼
客户端轮询 GET /api/analysis/result/{taskId}
（每 1 秒轮询，最多 30 次，超时返回错误提示）
```

## 五、AI 音频分析模块

### 5.1 模型选型详解

#### 基础分类模型：YAMNet（推荐）

- **来源**：Google Research，TensorFlow Hub 开源
- **特点**：预训练于 AudioSet（超 200 万音频样本），支持 521 类音频分类
- **微调策略**：
  1. 冻结底层特征提取层
  2. 替换顶层分类头（输出 8 类宠物情绪）
  3. 使用迁移学习，仅需约 **500-1000 条** 标注音频即可获得可用效果

#### 训练数据收集计划

| 阶段 | 数量目标 | 来源 |
|------|---------|------|
| MVP 冷启动 | 每类 50 条（共 400 条）| 团队自采 + 宠物博主合作 |
| 成长期 | 每类 500 条 | 用户众包标注（打卡奖励） |
| 成熟期 | 每类 5000+ 条 | 用户自动标注 + 专家审核 |

#### 众包标注交互设计

- 用户上传录音后，AI 给出初步判断
- 展示"你的毛孩子是这个意思吗？"确认弹窗
- 用户点击确认/纠正，即完成一次标注
- 纠正行为记录为高质量标注，优先入训练集

### 5.2 音频预处理流水线

```python
# audio_processor.py
import librosa
import noisereduce as nr
import numpy as np

class AudioProcessor:
    TARGET_SR = 16000      # 目标采样率
    MAX_DURATION = 15      # 最长音频秒数
    
    def process(self, file_path: str) -> dict:
        # 1. 加载音频
        audio, sr = librosa.load(file_path, sr=self.TARGET_SR, mono=True)
        
        # 2. 截断超长音频
        if len(audio) > self.TARGET_SR * self.MAX_DURATION:
            audio = audio[:self.TARGET_SR * self.MAX_DURATION]
        
        # 3. 降噪
        audio_denoised = nr.reduce_noise(y=audio, sr=sr)
        
        # 4. 有效音段检测（VAD）去除静音
        intervals = librosa.effects.split(audio_denoised, top_db=30)
        if len(intervals) == 0:
            raise ValueError("未检测到有效声音，请重新录制")
        
        # 5. 提取特征
        mfcc = librosa.feature.mfcc(y=audio_denoised, sr=sr, n_mfcc=40)
        mel_spec = librosa.feature.melspectrogram(y=audio_denoised, sr=sr)
        
        return {
            "audio": audio_denoised,
            "sr": sr,
            "mfcc": mfcc,
            "mel_spec": mel_spec,
            "duration": len(audio_denoised) / sr,
        }
```

### 5.3 模型推理服务

```python
# inference_service.py
from fastapi import FastAPI, UploadFile
from pydantic import BaseModel

class AnalysisResult(BaseModel):
    task_id: str
    pet_type: str           # "cat" | "dog"
    emotion: str            # 主要情绪
    confidence: float       # 置信度 0~1
    secondary_emotion: str  # 次要情绪
    health_alert: bool      # 是否触发健康预警
    raw_scores: dict        # 各情绪原始分数
    llm_description: str    # AI 生成的文案

app = FastAPI()

@app.post("/analyze")
async def analyze_audio(task_id: str, audio_file: UploadFile):
    processor = AudioProcessor()
    classifier = EmotionClassifier()  # 加载微调后的 YAMNet
    llm_client = LLMClient()
    
    # 预处理
    features = processor.process(await audio_file.read())
    
    # 推理
    scores = classifier.predict(features["mel_spec"])
    emotion = max(scores, key=scores.get)
    confidence = scores[emotion]
    
    # 置信度过低时触发规则引擎兜底
    if confidence < 0.5:
        emotion = rule_engine_fallback(features)
    
    # 健康预警检测
    health_alert = emotion == "uncomfortable" or confidence < 0.4
    
    # LLM 文案生成
    description = await llm_client.generate(
        emotion=emotion,
        confidence=confidence,
        pet_info=get_pet_info(task_id)
    )
    
    return AnalysisResult(
        task_id=task_id,
        emotion=emotion,
        confidence=confidence,
        health_alert=health_alert,
        llm_description=description,
        # ...
    )
```

## 八、安全与隐私

### 8.1 数据安全

| 措施 | 实现方式 |
|------|---------|
| 传输加密 | 全站 HTTPS (TLS 1.3)，小程序强制 HTTPS |
| 音频文件加密 | OSS 服务端加密（AES-256），临时 URL 访问（有效期 1 小时）|
| 数据库加密 | 敏感字段（手机号、openid）AES 加密存储 |
| SQL 注入防护 | 使用 ORM（Sequelize / Prisma），参数化查询 |
| XSS/CSRF | Koa 开启 csrf 中间件，参数校验使用 Joi |

### 8.2 接口安全

```
限流策略（Nginx + Redis）：
- 全局：每 IP 每秒 50 次
- 登录接口：每 IP 每分钟 10 次
- 分析接口：
  - 免费用户：每天 5 次
  - 月卡用户：每天 30 次
  - 年卡/永久：不限次
```

### 8.3 隐私合规

- 小程序首次启动展示**用户隐私保护协议**，需主动勾选同意
- 麦克风权限：仅在录音时请求，不常驻后台
- 音频文件：**分析完成后 7 天自动删除** OSS 中的原始文件
- 用户注销：支持一键注销，删除所有个人数据（符合《个人信息保护法》）
- 微信小程序需通过**微信隐私保护指引**审核

## 九、性能与监控

### 9.1 性能目标

| 指标 | 目标值 |
|------|-------|
| 分析接口响应（P99）| ≤ 5 秒 |
| 首页加载时间 | ≤ 1.5 秒 |
| 接口可用性 | ≥ 99.5% |
| 音频上传成功率 | ≥ 99% |

### 9.2 监控方案

```
应用监控：阿里云 ARMS / Sentry
接口监控：每分钟巡检核心接口
日志方案：Winston → 阿里云 SLS（日志服务）
告警规则：
  - 错误率 > 1% → 立即告警
  - P99 响应时间 > 3s → 告警
  - AI 服务 CPU > 80% → 扩容告警
业务监控（大盘）：
  - 每日活跃用户数（DAU）
  - 每日分析次数
  - 情绪分布统计
  - 用户转化率
```

### 9.3 扩容策略

- AI 分析服务部署于独立节点（GPU/CPU），根据队列积压自动扩容
- 数据库读写分离：主库写入，从库读取历史记录
- Redis Cluster 保障缓存高可用

## 十二、风险与对策

| 风险 | 概率 | 影响 | 对策 |
